# TenderMate — журнал решений

Хронология технических решений и почему они приняты именно так.
Правило: новый чат с Claude ВСЕГДА начинается с загрузки этого файла целиком.
Пиши в конце дня — что сделано, что решено, что осталось.

---

## 2026-06-23 — Миграция HTML → React

**Что:** Пересобрали прототип TenderMate с чистого HTML/JS на React 18 +
Vite + TypeScript + Tailwind. Стейт — только `useState` + localStorage.

**Почему:** Нужна была поддерживаемая структура для дальнейшего роста
продукта (компоненты, типы, хуки), а не монолитный HTML-файл. Redux/Context/
Zustand сознательно не взяли — на этом уровне сложности они не нужны,
лишняя зависимость и абстракция (KISS/YAGNI).

## 2026-06-23 — Дизайн-система: два раунда

**Что:** Раунд 1 — утвердили «Госархив» (тёмный фон + золото). Пользователь
заметил совпадение по ощущению с конкурентами и попросил пересмотреть.
Раунд 2 — утвердили «Бюро» (светлый фон, малахит `#0F766E`, Golos Text +
JetBrains Mono).

**Почему:** Прямые конкуренты (semantor.ru, neirotender.ru, gos-tender-ai.ru,
tenderai.pro) все используют белый фон + синий/индиго. «Бюро» — единственный
вариант из предложенных, где не задействован ни синий, ни красный, ни
золото (золото на тёмном ассоциируется с банкингом/крипто — Тинькофф,
Совкомбанк). Зафиксировали правило: белый+синий/индиго запрещён как копия
рынка.

## 2026-07-21 — Обнаружение геоблокировки Anthropic

**Что:** Запросы к Anthropic API с российских IP стали возвращать 403
Request not allowed.

**Почему это произошло:** экспортные ограничения США — подтверждено
официальным уведомлением Anthropic. Это не баг кода, а внешнее ограничение
инфраструктуры.

**Решение:** полный отказ от Anthropic в проде, миграция AI-слоя на
российского провайдера. Урок зафиксирован в `Описание.txt`: закладывать
РФ-совместимость с первого дня, не откладывать — миграция оказалась
трудоёмкой задним числом.

## 2026-07-21 — Выбор новой архитектуры: Timeweb + Yandex Cloud AI Studio

**Что:** Приняли стек — FastAPI на Timeweb App Platform (бэкенд),
Yandex Cloud AI Studio (OCR + LLM), Supabase временно оставлен только
как PostgreSQL.

**Почему:** Требование пользователя — вся инфраструктура обязана быть
РФ-совместимой без исключений. Timeweb — уже знакомая панель (фронтенд
там же), Yandex Cloud даёт и OCR, и LLM в одном месте с оплатой рублём.
Supabase не тронули сразу, чтобы не менять всё одновременно — минимизация
риска на один переезд за раз.

## 2026-07-21 — Асинхронная обработка через FastAPI BackgroundTasks

**Что:** Бэкенд переписан на FastAPI (`main.py`, `db.py`, `prompt.py`,
`llm.py`, `ocr.py`). Фоновая обработка — `BackgroundTasks`, статус —
поллинг фронтенда каждые 3 сек из таблицы `tender_jobs`.

**Почему:** Пробовали `EdgeRuntime.waitUntil()` в Supabase Edge Function —
не гарантирует завершения фоновой задачи после ответа на бесплатном
тарифе, проверено на практике (job зависал в статусе `processing`
навсегда). FastAPI BackgroundTasks на своём сервере — честно работает.

## 2026-07-21 — Баг: обрыв на пакетах из 4+ файлов

**Что:** `request.json()` падал на больших запросах — 4 PDF в base64
давали JSON ~2 МБ, парсер не справлялся.

**Почему упало:** base64 раздувает размер файла на треть, плюс весь
пакет уходил одним гигантским JSON-телом запроса.

**Решение:** переход на `multipart/form-data` — файлы идут потоком,
без base64 на входе. Изменены `main.py` (бэкенд) и `useAnalyze.ts`
(фронтенд).

## 2026-07-21 — Логика 44-ФЗ vs 223-ФЗ

**Что:** Промпт разделён на явные ветки по типу закупки.

**Почему:** Изначальный промпт писался под 44-ФЗ и ошибочно применял
антидемпинг (ст. 37, −25%) к закупкам по 223-ФЗ, где эта норма не
действует. Это давало пользователю неверную информацию о минимально
допустимой цене.

## 2026-07-21/22 — Поддержка Word/Excel, отказ от .doc

**Что:** Добавлены `.docx` (python-docx) и `.xlsx` (openpyxl) — прямое
чтение структуры, без OCR. Позже — `.xls` (xlrd 1.2.0). `.doc`
(Word 97-2003) сознательно НЕ поддержан.

**Почему:** Документация тендеров на практике чаще приходит в Word/Excel,
чем в PDF. Для `.doc` нет надёжного чистого Python-решения — нужен
LibreOffice как системная зависимость, а не просто pip-пакет; вместо
хрупкого полурабочего парсера решили явно сказать пользователю
«пересохраните в .docx», с точной инструкцией в сообщении об ошибке.

## 2026-07-22 — Итерации качества разбора по обратной связи на реальных тендерах

**Что:** Серия правок промпта по результатам ручной сверки с
документацией: «Состав заявки» перестал ссылаться на номера пунктов
(разворачивает в понятные формулировки); убран блок «Коридор для торга»;
добавлены предмет закупки/заказчик/место и срок поставки/условия оплаты
(с нормой закона по умолчанию — 7 раб. дней, если срок не указан явно);
добавлен `evaluationCriteria`; добавлены структурированные контакты
заказчика с копированием по кнопке; добавлен `nationalRegime` с явным
различением запрета/ограничения/преимущества по ПП 1875 простыми словами;
добавлено поле `isSmp` в профиль; добавлено обнаружение асимметрии
ответственности сторон в проекте контракта; добавлены ссылки на
нормативные акты в рисках.

**Почему:** Каждая правка — результат сверки реального вывода приложения
с реальной документацией закупки. «Коридор для торга» убран как
осознанное решение о границах продукта — TenderMate не берёт на себя
ценовую стратегию участника, это его зона ответственности. Национальный
режим объясняется по фиксированным шаблонам (не даём модели домысливать
юридически чувствительную тему).

## 2026-07-22 — Архитектурный пересмотр: двухэтапный пайплайн (extraction → synthesis)

**Что:** Разбор документа теперь идёт в два прохода: Extraction (по
каждому документу отдельно → факты с источником) → Synthesis (все
факты → финальный анализ).

**Почему:** Российские LLM имеют существенно меньшее контекстное окно,
чем Claude — отправка всех документов разом упиралась в лимит. Побочный
эффект оказался полезным: каждый факт теперь снабжён source (пункт/
раздел) — усилило Explainability по всему пайплайну, не только там,
где это специально требовалось.

## 2026-07-22 — Баг: лимит страниц Yandex Vision OCR

**Что:** Синхронный OCR API Yandex принимает 1 страницу за запрос —
многостраничные PDF падали с `Request pages N, service page limit is 1`.

**Решение:** `pypdf` разбивает PDF на отдельные страницы перед OCR,
результат склеивается после.

## 2026-07-22/23 — Критическая находка: ложный отказ LLM на юридическом тексте

**Что:** Yandex-модель отказывалась разбирать документы («Я не могу
обсуждать эту тему»).

**Почему это произошло:** формулировки состава/концентрации товара в
спецификациях («раствор кислот и щелочей 40%» для защитных перчаток,
«щёлочь 5%» для чистящих средств) — обычный язык описания товара,
ложно срабатывает фильтр безопасности модели. Проблема системная —
воспроизвелась на второй, не связанной с первой, закупке.

**Решение:** прямой структурный парсинг таблиц спецификации, полностью
минуя LLM (`spec_table.py`). Определение таблицы по ключевым словам в
заголовке колонок — не зависит от числа и порядка колонок, устойчиво
между разными закупками. Убирает риск отказа полностью для этого
класса документов, плюс дешевле и точнее (нет риска галлюцинации на
длинных списках).

## 2026-07-23 — Доводка парсера спецификации (три бага подряд)

**Что:** (1) вторая строка-заголовок с номерами колонок («1|2|3|4|5»)
попадала в данные как мусорная позиция — добавлена фильтрация;
(2) колонка «Цена за единицу» ложно матчилась как «единица измерения»
(обе содержат слово «единиц») — добавлен список исключающих слов;
(3) дедупликация по одному названию схлопывала разные товары
(например перчатки разных размеров) — ключ сравнения расширен до
«название + характеристики».

**Почему каждый раз чинили по факту, не заранее:** структура таблиц
спецификации сильно варьируется между закупками (разное число колонок,
разные формулировки заголовков) — только тестирование на реальных
документах вскрывало эти случаи, синтетических тестов было недостаточно.

## 2026-07-23 — Устойчивость к сбоям

**Что:** Текст ошибки никогда не бывает пустым (fallback на имя класса
исключения); имя документа добавлено в контекст ошибки на этапах чтения
файла и LLM-анализа; одна повторная попытка при подозрении на отказ
модели; сбой на одной картинке/странице не роняет весь разбор.

**Почему:** `str(e)` у некоторых исключений (например `httpx.ReadTimeout`)
даёт пустую строку — из-за этого пользователь видел неинформативный
fallback-текст вместо реальной причины сбоя. Без имени документа в
ошибке было невозможно диагностировать, какой из 5-6 файлов пакета
вызвал проблему.

## 2026-07-23 — UX: живой прогресс и скоринг сложности (частично не доехало до репозитория — см. ниже)

**Что:** Добавлена колонка `stage` в `tender_jobs` — бэкенд пишет
текст текущего этапа разбора. Добавлен клиентский скоринг сложности
заявки (`ComplexityBadge`). Логотип: «Tender» тоньше, «Mate» жирным.

**Почему:** Продукт сейчас в фазе «показать и продать» (решение
пользователя), не «отточить для постоянного пользователя» — приоритет
отдан тому, что видно в первые 60 секунд использования. Скоринг
сложности и логотип-задел под линейку — дешёвые изменения с быстрым
видимым эффектом, не требующие новой инфраструктуры.

**⚠️ Расхождение, найденное 23.07 в другом чате:** `ComplexityBadge.tsx`
в реальном фронтенд-репозитории НЕ существует — при деплое после правки
`ResultsView.tsx` сборка на Vercel упала с `Cannot find module
'./ComplexityBadge'`. Похоже, эта фича обсуждалась/писалась в одном из
чатов, но так и не была закоммичена в репозиторий (или удалена позже).
Импорт и использование убраны из `ResultsView.tsx`. **Урок:** запись в
журнале «добавлено» не гарантирует, что это реально в репозитории —
перед тем как опираться на прошлую фичу, стоит перепроверить факт её
наличия в актуальном коде (для публичного репо — `web_fetch`, для
приватного — попросить пользователя подтвердить/прислать файл).
Живой прогресс (`stage`) в БД — колонка была, но бэкенд её не заполнял
по ходу разбора вплоть до 23.07 (см. запись «Живой прогресс» ниже).

## 2026-07-23 — Перенос базы данных на Timeweb PostgreSQL: реализация

**Что:** База перенесена с Supabase на собственный PostgreSQL в Timeweb.
`db.py` переписан на `asyncpg` (был REST через `httpx` к PostgREST).
Интерфейс (`create_job`, `update_job`, `get_job`) сохранён — `main.py`
трогать не пришлось. Таблица `tender_jobs` создана вручную через
Adminer (веб-интерфейс Timeweb): `id uuid, status text, stage text,
result jsonb, error text, created_at/updated_at timestamptz`.

**Почему:** Supabase — последний оставшийся не-РФ сервис в цепочке после
переезда AI и бэкенда. Плюс практическая причина: бесплатный тариф
Supabase уходил в автопаузу при простое, что уже останавливало разбор
посреди работы.

**Баг инфраструктуры Timeweb (подтверждён поддержкой):** приватную сеть
нельзя подключить к уже существующему приложению App Platform задним
числом — работает только если выбрать сеть сразу при создании нового
приложения. Обходной путь на сейчас: публичный IP у базы + TLS
(`sslmode=verify-full`) + firewall-правило на TCP/5432. Диагностика
заняла много времени: `psql` с домашнего Мака зависал даже при открытом
порте (TCP отвечал, но полноценное соединение — нет) — похоже на
проблему MTU/фрагментации в домашней сети, не в самой базе; подтвердили
через веб-интерфейс Adminer (HTTPS/443 через браузер прошёл нормально).
Если Timeweb когда-нибудь починит баг — можно пересоздать App Platform
с сразу подключённой приватной сетью и выключить публичный IP у базы.

**Секреты сменились:** `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` →
`DATABASE_URL`. Anthropic-переменные удалены полностью (не используется
в проде вообще, см. запись про геоблокировку).

## 2026-07-23 — Баг лимита токенов на реальном тендере (104-позиционная таблица без «количества»)

**Что:** Разбор падал: `number of input tokens must be no more than
32768, got 42166`. Причина найдена точно — таблица технических
характеристик (104 позиции, 72к символов) не распознавалась как
спецификация в `spec_table.py`, потому что эвристика требовала
name+qty ОБЯЗАТЕЛЬНО, а в этой закупке колонки «количество» не было
вообще (рамочный договор — заказчик резервирует сумму, а объём
поставки определяется заявками по ходу исполнения). Вся таблица целиком
уходила в текст для LLM.

**Проверено:** 32768 токенов — лимит платформы Yandex AI Studio в целом
(не конкретной модели) — смена модели через `YANDEX_MODEL_URI` не решила
бы проблему.

**Решение:** `_detect_columns` теперь принимает name + (qty ИЛИ char),
не оба сразу. Плюс новая структурная (без LLM) проверка — если ни у
одной позиции спецификации нет `qty`, в `risks` добавляется
предупреждение «Вероятно рамочный договор» с объяснением для
пользователя.

## 2026-07-23 — Номер извещения, площадка, ссылка на закупку

**Что:** Три новые категории extraction, собираются напрямую из фактов
(как `customerContacts`), минуя synthesis: `procurementNumber`,
`platform`, `platformUrl`. На фронтенде — в блоке «Тип закупки»:
название площадки (ссылкой, если есть куда вести), номер извещения с
кнопкой копирования.

**Почему именно так:** Обсуждали вариант с автоматическим построением
ссылки на карточку закупки в ЕИС по реестровому номеру — отказались,
не смогли на 100% подтвердить точный формат URL, а показывать
непроверенную ссылку рискованно для доверия к продукту. Вместо этого:
`platformUrl` берётся ТОЛЬКО если реально написана в документе (проверено
на практике — у одной закупки на РТС-тендере номера извещения в тексте
не было вообще, а адрес площадки был) — не выдумывается по названию.
Резервный список — только 2 домена, оба подтверждены поиском
(`rts-tender.ru`, `sberbank-ast.ru`); B2B-Center не подтверждён
уверенно — оставлен просто текстом без ссылки. Для 44-ФЗ без прямой
ссылки — страница расширенного поиска ЕИС (подтверждённый рабочий URL,
не угаданная прямая карточка).

## 2026-07-23 — Живой прогресс наконец подключён от бэкенда до фронтенда

**Что:** `main.py` теперь пишет `stage` в БД на каждом шаге разбора:
«Читаю «...»…», «Ищу факты в «...»…», «Собираю итоговый анализ…».

**Почему это было отдельным багом:** колонка `stage` в БД и код на
фронтенде (`useAnalyze.ts`), который её читает и показывает
(`if (job.stage) setLoadingMsg(job.stage)`), существовали уже давно —
но бэкенд никогда не писал `stage` по ходу процесса, только в самом
конце (`complete`/`error`). То есть вся инфраструктура для «вау-эффекта»
живого прогресса была готова с обеих сторон и просто не была соединена.

## 2026-07-23 — Мелкая доводка: формат сумм

**Что:** НМЦК и другие суммы — только сокращение «руб.», без склонений
(«рубля»/«рублей» запрещены явным правилом в промпте synthesis).

**Почему:** Модель сама выбирала склонение непоследовательно
(«6 000 000,00 рубля»), выглядело неаккуратно и непрофессионально в

## 2026-07-23 — Процессные решения (работа с Claude)

- Присылать изменённые файлы ЦЕЛИКОМ, не диффами — пользователю неудобно
  собирать фрагменты вручную на GitHub. Заменяет прошлое правило
  «точечные правки предпочтительны».
- Лимит 250 строк на файл — жёсткое требование, соблюдать при каждой
  правке (несколько раз пришлось сокращать докстринги/комментарии,
  чтобы уложиться).
- Бэкенд-репозиторий публичный (misiknordic-design/TenderMate-backend)
  → Claude должен САМ читать актуальное состояние файлов через
  `web_fetch` перед правками, не полагаться на память чата — подтверждено
  на практике, что без этой перепроверки легко работать с устаревшей
  версией файла.
- Фронтенд-репозиторий приватный (misiknordic-design/TenderMate) →
  Claude НЕ видит его напрямую, полностью зависит от файлов, вставленных
  пользователем в чат. Архивы «из другого чата» ненадёжны — см. случай
  с `ComplexityBadge.tsx` выше.
- Заведён этот файл (`PROJECT_LOG.md`) как единый источник истории
  решений по всему продукту (не только бэкенду) — независимо от
  автоматической фоновой памяти Claude, которая может отставать/терять
  нюансы «почему» между отдельными чатами.
